# わんワード（犬用語カタカナ4文字パズル）実装プラン ver.4

> ver.2 変更点: 別セッション相当のレビュー指摘（判定ロジックのテスト粒度、JST日付関数の共有、streak仕様の穴、ゲーム難易度、アクセシビリティ等）を全面反映。加えてユーザーからの仕様変更（**候補46語をゲーム画面で公開する**・Base64難読化を廃止・キーボードを実使用文字のみに絞る）を反映した。
>
> ver.3 変更点: **キーボードのレイアウトを五十音表形式（縦5段×横9行）に変更**。ランダム配置ではなく学習コストゼロの五十音順にし、「行ごとにグレーが揃う」視認性を狙う。経由専用キー（ソ・ヒ・ヘ）も表の定位置に配置し穴を作らない。「ン」はワ行オ段（見慣れた五十音表通りの位置、かつ46語での頻出文字であるため）に配置し、補助キー行には回さない。Phase 3の完了条件に実機でのタップしやすさの確認を追加。
>
> ver.4 変更点: 判定ロジックのお手本トレースの誤りを訂正（正解「ワンコ」+入力「ンンワ」→ **グレー・緑・黄** が正。旧「黄・グレー・黄」はアンチパターンとして回帰テスト化）。キーボード網羅性テストを新設（46語の全文字が入力可能かを機械検証）。単語リスト46語を付録に平文で列挙。**ヤ行を五十音表に復元**（縦5段×横10行に訂正。ヤ・ユは空欄、ヨはヤ行オ段の定位置）。キーボードの寸法・固定表示・実機ゲートの合否基準を具体化。TZテストに制御アサーションを追加。JST絶対日数を一発計算式に統一し負のmodを正規化。localStorageの失敗耐性・スキーマversion・厳密な日付一致復元を仕様化。濁点キーの対象を「カーソル位置」として明確に定義。CSPインライン禁止の注意書きを追加。**出題ロジックを`dayNumber % 46`の逐次選出から、固定シードで並べ替えた索引表を引く方式に変更**（候補公開は維持しつつ、順番からの逆算だけを防ぐ）。反映箇所は末尾の「レビュー指摘・仕様変更の反映対応表」を参照。

## 概要

犬に関するカタカナ4文字の単語を6回以内に当てる、1日1問の静的パズルゲーム。**HTML/CSS/JavaScriptのみで動作する完全な静的サイト**として実装し、GitHub Pages で公開する。外部API・外部データソース・サーバーサイドコードは一切使わない。

デザイン・コーディング規約は既存の姉妹プロジェクト **`散歩危険度チェッカー`**（同デスクトップ内）に世界観を揃える。カラートークン、カード/ピルボタンのCSS変数、`node --test`によるテスト方針、`innerHTML`禁止のDOM描画方針、ES Modules構成、そして**色だけに頼らない表示（色+テキスト+アイコン併用）という設計原則**をそのまま踏襲する。

## ユーザー確定事項（このプランの前提）

- 文字種: カタカナのみ。「バ」「パ」は「ハ」と別の1文字。小さい文字（ャッェ等）・長音符「ー」も独立した1文字としてカウント
- 単語長: 4文字固定 / 挑戦回数: 6回
- 色分け: 緑(位置・文字とも一致) / 黄(文字は含むが位置違い) / グレー(含まない)。**色だけに依存せず、記号・テキストも併用する**
- 判定ロジックの処理順序（**最重要・逸脱不可**）:
  1. 全マスを走査し、位置も文字も一致するものを緑に確定。緑にした分だけ正解側の「その文字の在庫」を1減らす
  2. 残った入力文字を**左から順に**見て、在庫が残っていれば黄、在庫切れならグレー
- 入力: 画面上のカタカナキーボードのみ（物理キーボード入力・IME変換は対象外）。濁点キーで清音→濁音→（ハ行のみ）半濁音を循環
- 辞書チェックなし。カタカナ4文字が揃っていれば送信可能
- 1日1問。**日付の切り替えは日本時間(JST)固定・0:00**。端末のタイムゾーン設定に依存させず、UTC時刻から一発計算式でJSTの絶対日数を算出する
- **候補46語をゲーム画面から閲覧できるようにする**（折りたたみ or 別画面）。「今日の答えはこの46語のどれかです」と明示し、絞り込みという推理を成立させる
- **【ver.4変更】出題順は`dayNumber % 46`による単語リストの逐次選出ではなく、固定シードで並べ替えた索引表を引く方式にする**（候補プールの公開は維持しつつ、単語リストの並び順から今日・明日の答えを逆算できないようにする。理由: 順番まで読めてしまうとstreakの達成感が損なわれるため）
- **46語は46日で1巡する**。これは仕様として受け入れ、一巡後は単語を追加していく運用とする。「遊び方」に明記する
- `localStorage` に当日の結果と連続記録(streak)を保存
- 単語リストは**難読化しない**（候補を公開するため、Base64等の変換は不要かつ無意味）
- 「Wordle」という名称は商標のため、コード内・表示ともに一切使用しない
- デザイン: `散歩危険度チェッカー`と共通のアクセントカラー `#3d8f63`、カード角丸16px、ピルボタン999pxを踏襲

## 技術的に注意すべき落とし穴（実装前に共有しておく設計判断）

1. **Unicode正規化(NFC/NFD)**: 「バ」は通常1コードポイント(U+30D0)で表現される(NFC)が、環境によっては「ハ」+結合濁点(U+3099)の2コードポイント(NFD)になりうる。**全ての文字列比較の前に`normalize('NFC')`を通す**ことをコーディング規約としてCLAUDE.mdに明記し、判定ロジックのテストにも正規化前提の入力を混ぜて防御を確認する
2. **文字分割は`Array.from()`（またはスプレッド構文）を使う**: コードポイント単位分割を明示することで意図を固定する
3. **【ver.4修正】JST絶対日数は一発計算式に統一する**: `getJstEpochDayNumber(date) = Math.floor((date.getTime() + 9*3600*1000) / 86400000)`。Y/M/Dに分解してから組み立て直す実装はオフバイワンの余地を生むため行わない。**Y/M/Dは表示用途（画面に「2026年7月14日」等を出す場合）にのみ別関数で導出し、ゲームロジックの意思決定（出題インデックス選出・streak判定）には一切使わない**
4. **濁点サイクルの非対称性**: ハ行（ハヒフヘホ全て）は「清音→濁音→半濁音」の3状態循環、カ・サ・タ行は「清音→濁音」の2状態循環。それ以外の行・小さい文字・長音符には濁点対象がなく、キーを押しても無変化とする
5. **日付計算の一元化**: `day.js`に**JSTの絶対日数（エポックからの通し番号、mod前の生の値）を返す純粋関数を1つだけ**用意する。単語インデックス選出もstreak判定（前日か否かの差分計算）も、**必ずこの1つの関数だけを参照する**。`storage.js`が独自に「今日」「前日」を計算するコードを書くことは禁止（エイリアシング事故・TZバグの二重実装を防ぐため）
6. **色だけに依存しない表示**: タイル・キーボードの状態は色に加えて**記号（例: 緑=○、黄=△、グレー=✕）とスクリーンリーダー向けテキスト（`aria-label`）を併用**する。姉妹プロジェクトのリスクバッジ（色+アイコン+テキスト）と同じ設計原則
7. **キーボードは五十音表レイアウトに固定し、穴を作らない**: ランダム配置ではなく、五十音表（**縦5段×横10行**）に沿って配置することで、学習コストをゼロにし、かつ「行ごとに色が揃う」というWordle系ゲーム特有の面白さ（未使用文字を視覚的に探す）を成立させる。**経由専用キー（ソ・ヒ・ヘ）や、頻出だが1文字しか生き残っていない行（ヤ行のヨ）も、「省く」のではなく必ず表の定位置に配置する**（一部だけ穴になっていると探しにくくなるため。空欄を許容することと、行そのものを消すことは別物）。一方、46語にも経由地点にも一切使われない文字（テ・ナ・ラ等）は、その行の該当マスを素直に空欄のままにする（これは通常の五十音表でも見慣れた形であり、混乱を生まない）
8. **【ver.4新規】濁点キーが作用する対象は「カーソル位置」と定義する**: 「直前に追加した文字」を**盤面上で現在最後に埋まっているマス**と厳密に定義することで、空の状態（対象なし→無変化）・削除直後（カーソルは新たな最後尾マスに移る）・4文字目まで埋まった状態（カーソルは4マス目）のいずれでも、濁点キーの作用対象が一意に定まる
9. **【ver.4新規】負のmod対策**: 日数差分やインデックス計算で`n % 46`のようなJSの剰余演算をそのまま使うと、`n`が負の場合に負の値が返り無効な配列インデックスになりうる（例: 端末時刻が起点日より前）。**`normalizeMod(n, m) = ((n % m) + m) % m`を経由する**ことを規約とし、負数入力のテストを1本用意する
10. **【ver.4新規】localStorageの失敗耐性**: プライベートモード・ストレージ無効・容量超過・保存データの破損（JSON.parse失敗）は現実に起こりうる。**全てのlocalStorage操作をtry/catchで包み、保存に失敗してもゲーム自体はメモリ上の状態で遊べる**（streakや履歴だけが非永続化になる）というdegradation方針にする。スキーマに`version`フィールドを持たせ、バージョン不一致・パース失敗時は保存データを破棄して初期化する
11. **【ver.4新規】リロード復元は日付の完全一致を条件にする**: 保存されている`dayNumber`と、`day.js`から得た現在の`dayNumber`が**完全に一致する場合のみ**当日データを復元する。一致しなければ（日付が変わっていれば）古いデータは破棄し、その日の新規プレイとして開始する
12. **【ver.4新規】TZ切替テストの落とし穴**: Windows等の環境では、プロセス起動後に`process.env.TZ`を書き換えても`Date`の挙動に反映されない場合がある。**「TZ切替が実際に効いているか」を、既知のTZ依存操作（例:`new Date(0).getHours()`）で先に制御アサーションし、効いていなければテスト自体を明示的に失敗させる**（見せかけの「TZに依存しない」という誤った安心を防ぐ）
13. **【ver.4新規】出題順の秘匿には限界がある**: 固定シードによる並べ替え索引表も、静的サイトである以上ソースコードを実際に実行すれば将来の答えも計算できてしまう。これは「単語リストの並び順を見ただけで分かってしまう」という**うっかり見えるケースを防ぐもの**であり、完全な秘匿ではない。Base64のときと同じ誠実さの原則として、この限界をREADME/CLAUDE.mdに明記する
14. **CSPにインラインを混在させない**: `style=""`属性やインラインイベントハンドラ（`onclick=""`等）を使わない。CSSは`style.css`に、DOM操作は`sanitize.js`経由の`classList`/`createElement`で行う（Phase 1で設定するCSP `default-src 'self'`はインラインを許可しないため、混入すると動作しないか規約違反になる）

## 出題ロジックの設計変更（日付ハッシュ方式・ver.4）

- **問題意識**: `dayNumber % 46`で単語リストを順番通りに選ぶ設計だと、`words.js`の並び順を見るだけで「今日は何番目、明日は何番目」と誰でも逆算できてしまい、streak（連続記録）の達成感が薄れる
- **方針**: 候補46語のプール自体は引き続き画面で公開する（絞り込みという推理は維持）。変えるのは**「どの日にどの単語が割り当てられるか」という対応関係だけ**
- **実装方式**: 0〜45の並べ替え（permutation）を1つ、**固定シード付きの軽量PRNG（例: mulberry32等、32bit整数演算のみで浮動小数点誤差やエンジン差異が出ない方式）でFisher-Yatesシャッフルし、その結果を`day.js`にリテラル配列としてハードコードする**（実行時に毎回シャッフルを計算し直すのではなく、1度生成した固定表を埋め込む。生成に使ったシード値とアルゴリズムはコードコメントに残す）
- 単語インデックス = `WORD_ORDER_PERMUTATION[normalizeMod(getJstEpochDayNumber(date), 46)]`
- **決定論性**: PRNGは整数演算のみで浮動小数点を使わないため、同じシード・同じアルゴリズムなら常に同じ並べ替えが再現できる。端末やブラウザによって出題がズレることはない
- **テストでの担保**: `test/word-index.test.js`で、コメントに残したシード・アルゴリズムから並べ替え表を独立に再生成し、ハードコードされた`WORD_ORDER_PERMUTATION`と完全一致することを確認する（改変・コピペミスによる破損を検知する）
- **正直に明記する限界（設計判断13参照）**: この方式は「今日の答えを教えてしまう」ことを防ぐものではない（候補プールは公開されているし、ソースを実行すれば誰でも計算できる）。防ぐのはあくまで「単語リストの並び順を一瞥しただけで先読みできてしまう」という程度の話であることをREADME/CLAUDE.mdに明記する

## 進め方（重要）

このプランは **Phase 0 → Phase 6 を1つずつ実装 → 動作確認 → ユーザー承認 → 次のPhaseへ**、という順序で進める。**Phase 7（GitHubリモート作成・push・Pages有効化）は外部公開を伴うため、Phase 0〜6が完了し最終承認を得た後、実行直前に改めて明示的な承認を得てから行う**（自動連鎖では実行しない）。

このコマンド（`/plan`）では計画の作成のみを行い、実装は一切行わない。

  - `localStorage`に当日の日付情報（`day.js`の`getJstEpochDayNumber`から得た値のみを使用。独自の日付計算はしない）・単語インデックス・入力履歴・結果(`playing`/`win`/`lose`)を保存
  - **【ver.4新規】保存スキーマに`version`フィールドを持たせる**。読み込み時に`version`不一致、または`JSON.parse`が失敗する（データ破損）場合は、そのデータを破棄して初期化する
  - **【ver.4新規】全てのlocalStorage読み書きをtry/catchで包む**。プライベートモード・ストレージ無効・容量超過等で保存に失敗しても、ゲーム自体はメモリ上の状態で遊べる（streakや履歴だけが非永続化になる）というdegradation方針にする
  - **【ver.4明確化】当日データの復元は、保存されている`dayNumber`と現在の`dayNumber`が完全に一致する場合のみ行う**。一致しなければ（日付が変わっていれば）古いデータは破棄し、その日の新規プレイとして開始する
- streak管理の確定仕様:
  - 勝利時: 保存されている最終プレイ日の絶対日数が「今日−1」なら+1、そうでなければ1にリセット
  - 敗北時: 0にリセット
  - **同日内の多重加算防止（冪等性ガード）**: 最終プレイ日の絶対日数が既に「今日」と一致する場合、streak加算処理を再実行しない
  - **ページ読み込み時の即時補正**: 保存されているstreakが「今日−1」より古い（＝間に未プレイの日がある）場合、その日の結果を待たずに**その場でstreak表示を実態に合わせて0へ補正する**（古い数字をそのまま出し続けない）
  - **【ver.4新規】時計巻き戻りの扱い**: 保存されている`dayNumber`が現在の`dayNumber`より大きい（端末時計が過去に戻された等の異常）場合も、安全側としてstreakを0にリセットする
  - **最長記録(ベスト)はv1では保存しない**。現在のstreakのみ
- リロード時に当日分のデータがあれば盤面・結果表示を復元

**完了条件（検証ゲート）**
- `node --test`で`storage.js`の日付またぎ・streak更新ロジックを**相対的な日数差分**（絶対的な日付文字列ではなく`day.js`の絶対日数どうしの差分）でユニットテスト:
  - 前日プレイ→streak継続、2日以上の空白→表示が即座に0へ補正される、当日敗北→0
  - 同日内に勝利処理が2回呼ばれてもstreakが2重加算されないこと
  - **【ver.4新規】時計巻き戻り（保存dayNumber > 現在dayNumber）でstreakが0にリセットされること**
  - **【ver.4新規】localStorageが例外を投げる状況（モック）でもクラッシュせず、ゲームがプレイ可能な状態を維持すること**
  - **【ver.4新規】`version`フィールド不一致・JSON破損データを与えた場合に正しく初期化されること**
  - **【ver.4新規】保存`dayNumber`が現在の`dayNumber`と不一致（日付またぎ）の場合、古いデータが破棄されその日の新規プレイとして開始されること**
- Claude Previewでリロードしても当日の結果が復元されることを確認
- ユーザーが streak の確定仕様（敗北で0即時リセット、空白日は次回訪問時に即補正、時計巻き戻りで0、ベストは保存しない）に問題ないか承認
